## 작업 목표

핫스팟/CRT 오버레이의 색상·알파·글로우 강도가 기존 콘텐츠를 “너무 덮거나 너무 쨍하게” 만드는 문제를 해결하고, 기본 화면에서 더 보기 좋은 레이아웃/대비/반응형을 정리한다.

**배경**: `frontend/src/style.css`의 `.crt-overlay`(전역 스캔라인/플리커)와 `.hotspot-overlay`(hover/drag 강조)가 강하게 설정되어 있어, 콘텐츠 대비를 떨어뜨리거나 시야를 방해할 수 있다. 또한 오버레이 톤이 과도하면 “게임 UI”가 아니라 “필터를 씌운 화면”처럼 느껴져 데모 체감이 저하된다. (PRD 9.5, 리스크 R-011)

**완료 기준**:

- 핫스팟 기본/hover/drag 상태에서 오버레이가 Scene Canvas의 콘텐츠를 과도하게 덮지 않는다(알파/채도/글로우 튜닝).
- 전역 CRT 오버레이는 분위기는 유지하되, 기본 텍스트/패널의 대비를 떨어뜨리지 않는다(critical/ambient 레이어링 정책 유지).
- 모바일/태블릿 브레이크포인트(1200px/768px)에서도 오버레이/툴팁이 화면을 가리지 않고, 포커스/터치 타겟이 안전하다.
- 접근성: `prefers-reduced-motion: reduce`에서 플리커/펄스가 자동 완화되어 피로도를 낮춘다(기존 정책 유지/강화).

## 구현 흐름

### 1단계: 디자인 기준선 수립(스킬 활용)

- **frontend-design 스킬**로 “CRT 감성 유지 + 오버레이 덮임 최소화”를 목표로 한 팔레트/강도 개선안을 만든다.
- “오버레이가 강해야 하는 영역”과 “절대 가리면 안 되는 정보(critical)”를 체크한다(U-037 정책 준수).

### 2단계: CSS 토큰/값 튜닝(덮임/쨍함 완화)

- 핫스팟 오버레이:
  - 기본 상태 fill 알파를 낮추고(콘텐츠 보호), 강조는 border/outline 중심으로 이동한다.
  - hover/drag는 “네온 면적”이 커지지 않게 하고, 글로우는 짧고 얕게 유지한다.
- CRT 오버레이:
  - 스캔라인의 어두운 라인이 패널 텍스트 대비를 깎지 않게 강도를 조정한다.
  - reduced-motion에서 플리커를 더 강하게 완화한다(필요 시 애니메이션 빈도/opacity 범위 조정).

### 3단계: 가이드 기반 점검(스킬 활용)

- **web-design-guidelines 스킬**로 다음을 빠르게 점검하고, 필요 조정한다:
  - 대비/가독성(특히 작은 텍스트/툴팁)
  - 포커스 링/키보드 내비게이션(핫스팟 focus 포함)
  - 모바일 터치 타겟/툴팁 가림/화면 외부로 튀는 문제

### 4단계: 수동 검증(현실 시나리오)

- Scene Canvas 이미지 위에서 핫스팟이 “보이되 방해하지 않는지” 확인한다.
- 드래그 오버(드롭 타겟) 상태가 경고/가이드 역할을 하지만 화면을 과도하게 덮지 않는지 확인한다.

## 주의사항

**기술적 고려사항**:

- 오버레이는 항상 `pointer-events: none`을 유지하여 상호작용을 방해하지 않아야 한다(특히 드래그/클릭).
- 색상 변화는 CSS 변수/토큰 중심으로 수행해, 컴포넌트별 임의 색상 도입을 피한다(SSOT 유지).

**잠재적 리스크**:

- 오버레이를 너무 약하게 만들면 CRT 정체성이 약해질 수 있음 → “면적/강도”를 줄이되, Scene Canvas/크롬 영역에서 분위기를 확보한다.
